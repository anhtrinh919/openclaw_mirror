## 1) Where the “old session” lives
OpenClaw persists **session transcripts** on disk as JSONL, regardless of resets:

- Session store: `~/.openclaw/agents/<agentId>/sessions/sessions.json`
- Transcript files: `~/.openclaw/agents/<agentId>/sessions/<SessionId>.jsonl`

Docs: OpenClaw “Session Management” + transcript paths. (See `/usr/local/lib/node_modules/openclaw/docs/concepts/session.md`)

So when you `/new` or `/reset`, you’re starting a *new session context*, but the previous transcript file is still there.

## 2) How daily logs get generated in *our* setup
In this workspace, daily logs are primarily generated by our **scheduled cron job**:

- **Cron:** `daily-memory-review` (runs **23:59 Asia/Ho_Chi_Minh**)
- It scans recent sessions (including ones created earlier that day from `/new`/`/reset`)
- Then it **writes summaries** into:
  - `memory/YYYY-MM-DD.md` (daily log)
  - and curated items into `MEMORY.md`

So it’s not “moving” anything—it's a nightly summarizer that reads transcripts and writes memory notes.

(Separately, OpenClaw also supports an **automatic memory flush** when a session nears compaction—this can remind the agent to write to `memory/YYYY-MM-DD.md` before context gets compacted. Docs: `/usr/local/lib/node_modules/openclaw/docs/concepts/memory.md`.)

## 3) How it handles multiple resets in the same day
Each reset creates a different `SessionId` → a different `*.jsonl` transcript.  
The nightly memory job just aggregates across whatever transcripts exist for that day/time window and appends a combined daily summary.
